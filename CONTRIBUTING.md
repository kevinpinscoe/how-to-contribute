# How to Contribute

Thank you for taking the time to contribute. This guide applies to the
open-source projects maintained by Kevin P. Inscoe. Individual projects may
link here instead of repeating these instructions.

## Ways to Contribute

- **Report a bug** — open an issue describing what happened and what you expected.
- **Suggest a feature** — open an issue explaining the problem you want solved.
- **Improve documentation** — fix typos, clarify wording, or add missing detail.
- **Submit code** — fix a bug or implement a feature via a pull request.

## Reporting Issues

Before opening a new issue:

1. Search existing issues to avoid duplicates.
2. Use a clear, specific title.
3. Include steps to reproduce, the expected result, and the actual result.
4. Note your environment (OS, version, relevant tool versions) where it matters.

## Proposing Changes (Pull Requests)

1. **Fork** the repository and create a branch from `main`.
2. **Name the branch** for the work it contains, for example:
   - `fix/health-check-timeout`
   - `feat/add-backup-validation`
   - `docs/clarify-setup-steps`
3. **Make focused commits** — keep each commit centered on one related change.
4. **Write clear commit messages** using the `<type>: <short description>` style:
   ```text
   feat: add backup validation script
   fix: correct container health check
   docs: add deployment runbook
   ```
   Common types: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`.
5. **Test your change** and describe how you verified it in the pull request.
6. **Open the pull request** against `main` with a description of what changed
   and why.

## Code Style

- Prefer readable code over clever code.
- Add comments only where they clarify intent.
- Include basic error handling.
- Follow the existing style of the project you are contributing to.

## Review and Merge

- A maintainer will review your pull request and may request changes.
- Keep the discussion focused and respectful.
- Once approved, a maintainer will merge the change.

## Code of Conduct

Be respectful and constructive. Harassment, personal attacks, and
discriminatory language are not welcome in any project space.
